# Q. PODE 인스턴스들이 포함된 netlist로부터 모델 추출하는 법

최신 공정을 사용하고 있는데, 파운드리 PDK가 스키매틱에서 netlist를 추출할때 스키매틱 상에는 없는 추가 인스턴스들을 만들어내곤 합니다. 예를 들면, 트랜지스터 인스턴스에 정의된 옵션에 따라서, 하나의 트랜지스터가 두 개의 인스턴스로 넷리스팅될 수가 있습니다. 하나의 인스턴스는 트랜지스터 자체를 표현하고, 다른 하나의 인스턴스는 디퓨션 경계의 폴리실리콘 구조(poly over diffusion edge; PODE)와 관계된 RC 기생성분을 표현합니다. 아래의 예를 참고하세요. 이 경우, PDK netlister가 하나의 트랜지스터를 `'M1'`과 `'M1__macro'`라는 이름의 두개의 인스턴스의 조합으로 넷리스팅했는데요, 이 둘을 연결하기 위해 `'A_dmy'`, `'B_dmy'` 및 `'C_dmy'`와 같은 net들도 새로 만들었습니다. 이중 `'M1_macro'` 인스턴스는 트랜지스터의 드레인, 게이트, 소스 단자에 존재하는 추가 저항 및 커패시터 성분들을 표현하고 있습니다.

```
M1 ( A_dmy  B_dmy  C_dmy  D ) nch_svt_mac l=0.008u w=0.308u …
M1__macro ( A  B  C  D  A_dmy  B_dmy  C_dmy  D ) nch_svt_mac_pode l=72n …
```

문제는, 스키매틱 상에서는 하나의 인스턴스가 netlist 상에서는 복수의 인스턴스들로 변환되면서, MODELZEN 모델 추출과정에서 여러가지 문제를 발생시킨다는 것입니다. 예를 들어, 트랜지스터들을 묶어 하나의 UDM으로 설정할때, 코어 트랜지스터 인스턴스들만 UDM으로 묶이고, 추가로 생성된 PODE 인스턴스들은 안 묶이면서 모델 생성과정에서 오류들이 발생합니다. MODELZEN이 이렇게 변환된 복수의 인스턴스들을 하나의 묶음으로 보게 하는 방법이 있을까요? 그렇게만 할 수 있다면 모든 문제가 해결될 것 같은데요.

## Answers

### A1

정확하게 짚으셨네요! MODELZEN이 그 인스턴스들을 하나로 묶어 처리할 수 있도록 해주면 문제가 해결됩니다. 결론부터 말씀드리자면, technology configuration 파일의 `'devo_devicemap'`에 나열된 소자 매핑 항목들에 `'group'` 속성을 정의하시면 됩니다.
예를 들어, 아래와 같은 패턴으로 netlist된 두 개의 인스턴스들을 MODELZEN이 하나의 트랜지스터로 취급하게끔 하려는 경우를 가정해봅시다. 즉, `'nch_svt_mac'`와 `'nch_svt_mac_pode'` 인스턴스 쌍이 아래와 같이 외부 net인 `'A'`, `'B'`, `'C'`, 그리고 `'D'`에 연결되어 있는 경우에:

```
M1 ( A_dmy  B_dmy  C_dmy  D ) nch_svt_mac l=0.008u w=0.308u …
M1__macro ( A  B  C  D  A_dmy  B_dmy  C_dmy  D ) nch_svt_mac_pode l=72n …
```

이들을 아래에 표현한 것처럼 드레인(d), 게이트(g), 소스(s), 벌크(b) 터미널들이 각각 `'A'`, `'B'`, `'C'`, 그리고 `'D'`에 연결된 하나의 `'nmosfet'` 인스턴스로 변환되도록 하고 싶은 경우를 생각해봅시다.

```
nmosfet  M1 (.d(A), .g(B), .s(C), .b(D));
```

이에 해당하는 그룹핑 규칙을 technology configuration 파일에 기술하는 방법을 설명드리겠습니다. 편의를 위해 먼저 하나의 그룹으로 묶을 인스턴스들의 이름 패턴과 터미널 리스트를 정의하는 dictionary 변수를 정의하세요. 아래는 위의 예제에 대한 규칙을 `'group_pode'`라는 이름의 dictionary 변수로 정의한 것입니다.

```
# group property to merge transistor instances with their PODE instances
group_pode = {
    "@(INSTNAME)"           : dict(terms=['*', '*', '*', 'b']),
    "@(INSTNAME)__macro"    : dict(terms=['d', 'g', 's', 'b', '*', '*', '*']),
}
```

이것이 기술하고 있는 규칙은 다음과 같습니다: 만일 트랜지스터 인스턴스의 이름에 `'__macro'` 접미사가 붙은 또다른 인스턴스가 발견된다면, 그것들을 묶어 하나의 트랜지스터로 취급하되, 그것의 드레인, 게이트, 소스, 벌크 터미널은 인스턴스의 터미널 리스트에서 `'d'`, `'g'`, `'s'`, `'b'` 위치에 나열된 net에 각각 연결한다. 여기서, `"@(INSTNAME)"`은 embedded Python 표현식인데, `'INSTNAME'`은 대상 인스턴스의 이름을 가진 변수입니다. 이 embedded Python 표현식을 사용하면, 다양한 이름 패턴들을 쉽게 기술할 수 있습니다. 예를 들어, `"X@(INSTNAME[1:])"`는 첫글자가 `'X'`로 바뀐 인스턴스 이름을 표현하고, `"@(INSTNAME.upper())"`는 모든 영문자가 대문자로 바뀐 이름을 뜻합니다. 그리고, 터미널 리스트에 있는 별표(`'*'`)는 인스턴스들 사이를 연결하는 내부 net임을 의미합니다.
다음에는, 이렇게 정의한 dictionary 변수를 technology configuration 파일의 `'devo_devicemap'`에 나열된 소자 매핑 항목들의 'group' 속성값으로 사용하시면 됩니다. 아래는, 앞에서 정의한 `'group_pode'` dictionary를 `'nmos_svt_mac'`, `'pmos_svt_mac'`, `'nmos_lvt_mac'`, 그리고 `'pmos_lvt_mac'` 소자의 `'group'` 속성으로 지정함으로써, 이들 소자의 인스턴스가 netlist 상에서 위에서 말한 이름 패턴을 가진 PODE 인스턴스와 함께 발견되면 그것들이 하나의 그룹으로 취급되도록 설정하는 예를 보여줍니다.

```
# global device mapping
devo_devicemap = {
    "nmos_svt_mac"  : dict(type="nmosfet", vmax=1.2, max_leveldc=3, group=group_pode),
    "pmos_svt_mac"  : dict(type="pmosfet", vmax=1.2, max_leveldc=3, group=group_pode),
    "nmos_lvt_mac"  : dict(type="nmosfet", vmax=1.2, max_leveldc=3, group=group_pode),
    "pmos_lvt_mac"  : dict(type="pmosfet", vmax=1.2, max_leveldc=3, group=group_pode),
}
```

마지막으로, technology configuration 파일의 `'dev_model'` 항목에, 이들 PODE 소자들의 정의가 담긴 SPICE 모델 라이브러리의 위치를 정의하는 라인이 포함되어 있는지 확인하세요. 예를 들면, `'nch_svt_mac_pode'`의 subcircuit 정의가 들어있는 파일의 위치가 SPICE의 `.include` 또는 `.lib` 구문, 또는 Spectre의 `include` 구문으로 정의되고 있어야 합니다. 이는 MODELZEN이 이제 이 두 인스턴스들을 하나의 트랜지스터로 취급해 그 특성을 추출하는 SPICE 또는 Spectre 시뮬레이션을 수행할 것이기 때문입니다. 그 모델 라이브러리의 위치가 정의되지 않으면, MODELZEN이 SPICE 또는 Spectre 시뮬레이션을 수행하는 단계에서 *'해당 모델 또는 subcircuit의 정의를 찾을 수 없다'*라는 에러메시지를 내고 실행을 중단할 수 있습니다.
